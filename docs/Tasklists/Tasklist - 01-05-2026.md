## 1. Novo chat deve limpar cache da sessão

Status atual: concluído pelos commits `dc02f33`, `ba826c7` e documentação em `docs/frontend/NOVO-CHAT.md`.

### Objetivo

Ao apertar em **novo chat**, o app deve iniciar uma sessão realmente limpa, evitando vazamento de contexto, cache antigo, estado de threads anteriores ou resultados de agentes já concluídos.

### Subtasks

* [x] Investigar o fluxo atual do botão `novo chat`.
* [x] Identificar quais estados são limpos atualmente.
* [x] Identificar quais estados permanecem indevidamente após criar novo chat.
* [x] Mapear todos os caches envolvidos na sessão atual.
* [x] Verificar se existem caches separados para:

  * [x] mensagens;
  * [x] terminal output;
  * [x] threads;
  * [x] orquestrador;
  * [x] subagentes;
  * [x] contexto explícito;
  * [x] session ids nativos;
  * [x] provider ativo;
  * [x] resultados pendentes;
  * [x] histórico temporário;
  * [x] estado visual da UI.
* [x] Definir o que deve ser apagado em `novo chat`.
* [x] Definir o que não deve ser apagado, como configurações globais, providers cadastrados e preferências do usuário.
* [x] Implementar limpeza controlada de estado.
* [x] Garantir que threads antigas não reapareçam no novo chat.
* [x] Garantir que o orquestrador não receba contexto da sessão anterior.
* [x] Garantir que subagentes pendentes sejam encerrados, cancelados ou desassociados corretamente.
* [x] Documentar o comportamento esperado de `novo chat`.

### Possíveis problemas

* Estado de UI pode limpar, mas estado do processo pode continuar vivo.
* Sessões persistentes podem continuar com contexto antigo se não forem reiniciadas ou isoladas.
* Threads antigas podem continuar aparecendo se estiverem salvas em store global.
* Um subagente em execução pode terminar depois do novo chat e inserir resultado na sessão errada.
* Limpar tudo agressivamente pode apagar configurações que deveriam permanecer.

### Critérios de aceite

* [x] Ao criar novo chat, a tela começa sem mensagens antigas.
* [x] O terminal não mostra threads antigas como parte da nova sessão.
* [x] O orquestrador não recebe histórico anterior sem intenção explícita.
* [x] Subagentes antigos não conseguem injetar resultado no novo chat.
* [x] Configurações globais continuam preservadas.
* [x] O comportamento está documentado em `/docs`.

## 3. Orquestrador deve conhecer modelos disponíveis e limites de uso

Limites de custo estimado e contexto (tokens) adicionados.

### Objetivo

### Subtasks

* [x] Mapear providers/modelos atualmente disponíveis no app.
* [x] Identificar quais metadados já existem para cada modelo.
* [x] Criar ou atualizar uma estrutura de capacidades dos modelos.
* [x] Registrar informações como:

  * [x] nome do modelo/provider;
  * [x] tipo de execução: persistente, one-shot, resume, ACP, stream-json, app-server etc.;
  * [x] se suporta ferramentas;
  * [x] se suporta MCP;
  * [x] se suporta edição de arquivos;
  * [x] se suporta contexto longo;
  * [x] se suporta sessão nativa;
  * [x] se é mais indicado para código, revisão, resumo, planejamento ou escrita;
  * [x] limite de uso conhecido ou configurado;
  * [x] custo estimado, quando aplicável;
  * [x] prioridade/preferência do usuário;
  * [x] estado atual: disponível, indisponível, erro, sem login, limite atingido.
* [x] Definir como essas informações serão passadas ao orquestrador como contexto.
* [x] Evitar enviar contexto excessivo ou confuso ao orquestrador.
* [x] Permitir que o orquestrador explique por que escolheu determinado agente, quando necessário.
* [x] Registrar no log qual modelo foi escolhido e por quê, quando possível.
* [x] Documentar o formato do contexto de modelos.

### Possíveis problemas

* O orquestrador pode escolher modelos indisponíveis se o contexto estiver desatualizado.
* Informações demais podem aumentar custo/contexto sem melhorar decisão.
* Modelos podem mudar de comportamento com atualizações externas.
* Custo e limite podem ser estimativas, não valores exatos.

### Critérios de aceite
